# 观察者模式

​	**在嵌入式C语言开发中，观察者模式（Observer Pattern）是一种极其实用且经典的设计模式。它主要用于解决“一个事件触发多个后续动作”的场景，能显著提升代码的扩展性和可维护性。**



## 一、核心定义

​	观察者模式定义了一种**一对多**的依赖关系。当一个对象（称为“主题”）的状态发生改变时，所有依赖于它的对象（称为“观察者”）都会收到通知并自动更新。

### 1、核心角色

* **主题**：这是被观察的对象，比如一个按键或串口外设。它需要维护一个观察者列表，并提供注册和注销的接口。当自身状态变化时（如按键被按下），它会遍历观察者列表，通知所有观察者。
* **观察者**：这是依赖主题的对象，比如LED模块、日志模块。它需要提供一个统一的**回调函数**，主题通过调用这个函数来通知事件的发生。

### 2、工作流程

* **注册观察者**：主题注册观察者列表，知道有哪些观察者。
* **通知观察者**：在主题状态发生改变时，遍历所注册的观察者列表，通知所以观察者主题的状态发生改变。
* **观察者动作**：在接受到主题发放的通知后，执行对应动作与功能。

### 3、观察者模式的优势

* **低耦合 (Loose Coupling)**
  - **核心优势**：这是观察者模式最大的价值。主题只需要知道观察者实现了某个接口（或函数指针签名），而不需要知道观察者的具体类或实现细节。
  - **嵌入式场景**：例如，温度传感器驱动（主题）不需要包含 LCD 驱动或 WiFi 驱动（观察者）的头文件。传感器只管读数，LCD 和 WiFi 模块自己决定如何处理这个读数。这使得模块可以独立编译、独立测试。
* **符合“开闭原则” (Open/Closed Principle)**
  - **易于扩展**：当你需要增加一个新的功能（例如增加一个“蜂鸣器报警”模块）时，你只需要编写一个新的观察者函数并注册进去，而**不需要修改**现有的主题代码。
  - **嵌入式场景**：产品迭代时，如果要在原有的数据采集基础上增加“本地SD卡存储”功能，只需添加一个SD卡写入的回调函数并注册，无需改动核心的传感器采集逻辑，降低了引入新Bug的风险。
* **支持广播通信 (Broadcast Communication)**
  - **一对多通知**：主题状态改变一次，就可以自动通知所有注册的观察者。
  - **嵌入式场景**：当系统状态从“待机”变为“运行”时，可以一次性通知 UI 更新界面、电机开始转动、传感器开始采样，而不需要在主循环里写一堆 `if(status==RUN) { ... }` 的散弹式代码。
* **动态管理依赖**
  - **运行时灵活**：观察者可以在运行时动态地注册或注销。
  - **嵌入式场景**：在调试模式下，可以注册一个“串口打印调试信息”的观察者；在正式发布版本中，可以通过条件编译或运行时配置不注册它，从而在不修改核心代码的情况下控制功能。

### 4、观察者模式的缺点

* **执行顺序不确定 (Unpredictable Order)**

  * *潜在风险**：除非你在代码中明确规定了遍历数组的顺序，否则观察者被调用的顺序通常是不确定的（或者依赖于注册顺序）。

  - **嵌入式场景**：如果观察者B的处理逻辑依赖于观察者A已经执行完毕（例如：A负责解析数据，B负责显示数据），那么简单的观察者模式就会导致竞态条件或逻辑错误。**解决办法**：通常需要将强依赖的逻辑合并，或者使用责任链模式。

* **性能开销与实时性风险 (Performance Overhead)**

  - **同步阻塞**：在基础的C语言实现中，主题调用观察者通常是**同步**的。如果有一个观察者执行了耗时操作（比如通过I2C慢速写入EEPROM），它会阻塞主题和其他观察者的执行。

  - **嵌入式场景**：如果在中断服务程序（ISR）中直接触发观察者通知，而观察者里又有耗时操作，会严重阻塞中断，导致系统丢包或看门狗复位。

* **内存泄漏与悬空指针 (Memory Leaks & Dangling Pointers)**

  - **生命周期管理难**：在C语言中，如果观察者是一个动态分配的对象（或动态库），当它被销毁（free）后，如果忘记从主题的列表中移除（注销），主题持有的函数指针就会变成“悬空指针”。

  - **嵌入式场景**：一旦主题再次尝试调用这个已销毁的观察者，系统会立即发生 Hard Fault（硬件异常）并崩溃。这在长期运行的嵌入式设备中是致命的隐患。

* **调试困难 (Debugging Complexity)**

  - **调用链不直观**：数据流向变得隐蔽。你看到主题的状态变了，但很难一眼看出具体触发了哪些后续动作，特别是当观察者之间还存在级联通知时（A变了通知B，B变了通知C）。

  - **嵌入式场景**：当系统出现死机或逻辑错误时，追踪调用堆栈（Stack Trace）可能会发现是在某个回调函数里出的问题，但很难快速定位是哪个模块注册的这个回调。

### 5、使用建议

在以下场景可使用观察者模式来解决问题：

* **“一对多”的联动场景**

  - **场景**：一个硬件状态变化需要触发多个互不相关的动作。

  - **例子**：电池电压检测。当电压低时，需要：1. 点亮LED指示灯；2. 在屏幕显示警告图标；3. 通过蓝牙向手机发送通知。这三个模块互不干扰，都依赖电压数据。

* **模块间需要彻底解耦**

  - **场景**：你正在编写一个核心驱动（如传感器驱动），但你不想在这个驱动里包含任何业务逻辑（如UI、网络、存储）的头文件。

  - **例子**：编写通用的 `按键驱动`。驱动只负责识别“按下”、“长按”、“双击”事件，不负责具体功能。业务层（如“音量+”、“菜单确认”）作为观察者去订阅按键事件。

* **动态配置需求**

  - **场景**：系统功能需要根据配置开启或关闭。

  - **例子**：**调试日志系统**。你可以定义一个 `LogObserver`，在调试版本中将其注册到关键模块（如网络栈、文件系统），在发布版本中不注册，从而实现零侵入式的调试。

* **替代全局变量和标志位**

  - **场景**：代码中充斥着 `extern bool g_data_ready` 这种全局标志位，且多个地方都在轮询这个标志位。

  - **建议**：用观察者模式替代。数据产生者作为主题，消费者作为观察者，变“被动轮询”为“主动通知”。



在以下场景不推荐使用观察者模式来解决问题：

* **强依赖与顺序敏感**

  - **情况**：模块B的执行**必须**在模块A之后，且依赖A的处理结果。

  - **原因**：观察者模式通常不保证通知顺序（或者保证顺序很麻烦）。

  - **替代**：直接使用函数调用，或使用**责任链模式**。

* **实时性要求极高的关键路径**

  - **情况**：电机控制环路、高频信号采样等微秒级任务。

  - **原因**：函数指针调用、链表遍历都会引入不可预测的指令周期开销。

  - **替代**：直接内联代码或硬编码逻辑，性能第一。

* **观察者数量极少且固定**

  - **情况**：永远只有1个或2个固定的接收者，且未来几乎不可能增加。

  - **原因**：引入模式会增加代码复杂度（结构体、接口定义），属于过度设计。直接调用即可。

* **循环依赖风险**

  - **情况**：A通知B，B处理完又通知A。

  - **后果**：会导致死循环或栈溢出。设计时必须理清数据流向，确保是单向的。



### 6、实现建议

在使用观察者模式时需注意以下几点问题：

* **必须提供“注销”机制**

  - **规则**：有 `attach` 就必须有 `detach`。

  - **理由**：在C语言中，如果观察者是一个局部变量（在栈上）或者动态分配的内存，一旦它被销毁，主题里保存的指针就会变成**悬空指针**。下次调用时，系统会直接崩溃（Hard Fault）。

  - **做法**：在观察者模块的 `DeInit()` 函数中，第一件事就是调用主题的 `Detach()`。

* **严禁在ISR（中断）中直接通知**

  - **规则**：中断服务函数中只能发送信号量、释放互斥锁或写入消息队列，**绝对不能**遍历链表调用回调函数。

  - **理由**：观察者处理逻辑耗时不可控，会长时间占用中断，导致系统实时性崩塌。

  - **做法**：ISR -> 发送消息到队列 -> 任务A从队列读取 -> 任务A调用 `Notify`。

* **回调函数必须“快进快出”**

  - **规则**：观察者回调函数里只做“轻量级”操作（如置标志位、拷贝数据、发送消息）。

  - **理由**：因为是同步调用，一个慢的观察者会阻塞所有其他观察者。

  - **做法**：如果业务逻辑很重（如写入Flash），请在回调中创建一个新任务或在回调里只 `xQueueSend`。

* **静态内存优先**

  - **规则**：在资源受限的单片机中，尽量使用**静态数组**来存储观察者列表，而不是 `malloc` 链表。

  - **理由**：避免内存碎片，且静态数组访问效率更高，确定性更强。

* **增加“上下文”参数**

  - **建议**：回调函数定义最好带一个 `void* user_data` 或 `void* context` 参数。

  - **理由**：这样观察者函数才能知道是哪个具体的实例触发了通知（特别是在有多个同类传感器时），也能通过该指针访问观察者自身的结构体成员，避免使用全局变量。



# 总结

​	**在嵌入式C中，用观察者模式处理低频、非实时、多模块联动的事件（如UI交互、系统状态机、配置变更），但在高频、实时、强依赖的控制环路中，请坚持使用直接的函数调用。**
