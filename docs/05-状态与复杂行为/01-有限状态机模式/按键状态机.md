## 三、状态机群

### 1、并联式状态机

当我们需要处理的任务之间**没有相关性**（即彼此独立），但都需要进行时，可以将每一个任务抽象成一个独立的状态机。将复杂的问题拆分为多个简单的问题，这些简单的问题分别独立运行，没有互相影响且同时运行。

* **应用场景：**适用于多任务并发且任务间无数据传递或控制依赖的情况。

* **优势：** 极大地简化了单个状态机的复杂度，避免了在一个状态机中处理所有逻辑导致的“状态爆炸”。

### 2、串联式状态机

这种模式与并联式类似，内部的所有状态机仍然是**同时工作**的，但不同之处在于它们之间存在**数据传递**。

- **工作原理：** 前一级状态机的输出数据可能作为后一级状态机的输入，或者彼此之间存在数据上的依赖。虽然有数据关联，但从功能控制上看，它们依然是相对独立的，并非严格的前一个结束后后一个才开始。
- **适用场景：** 适用于任务之间不仅并发，而且存在明显的数据流转或处理流水线关系的场景。

### 3、串行式状态机

如果多件事情之间有**明显的先后顺序**且**不能并发**，就需要用到串行式状态机群。

- **工作原理：** 为每一件事情分配一个独立的状态机。前一级状态机在工作结束后，会向下一级状态机发送**控制信号**（如“开始工作”信号），下一级状态机收到信号后才开始工作。
- **适用场景：** 适用于复杂的流程控制，其中每个步骤本身很复杂（需要独立的状态机来描述），且步骤之间必须严格按顺序执行。



### 4、分层状态机

分层状态机（Hierarchical State Machine, HSM 或 HFSM）是为了解决传统扁平状态机在复杂系统中导致的“代码爆炸”和逻辑混乱问题而提出的。

- 核心结构：
  - **父状态（Super State）：** 包含多个子状态的高级状态，用于封装公共逻辑（如进入/退出动作、通用事件处理）。
  - **子状态（Sub-State）：** 嵌套在父状态内部，继承父状态的行为，并实现特有的差异化逻辑。
- 关键机制：
  - **行为继承：** 子状态可以直接利用父状态定义的通用逻辑，减少代码重复。
  - **事件冒泡：** 当事件发生时，先由当前子状态处理；如果子状态不处理，事件会“冒泡”传递给父状态处理。
  - **层级转换：** 支持父状态之间的切换，也支持子状态内部的独立切换。
- **适用场景：** 极其复杂的业务逻辑。

### 对比

| 模式名称   | 核心特征   | 任务关系           | 交互方式             | 典型应用                           |
| ---------- | ---------- | ------------------ | -------------------- | ---------------------------------- |
| 并联式     | 独立并行   | 无相关性           | 无交互，各自独立运行 | 按键扫描与LED显示同时进行          |
| 串联式     | 数据流水线 | 并发但有数据依赖   | 数据传递             | 数据处理流水线，前级输出供后级使用 |
| 串行式     | 严格顺序   | 先后执行，不可并发 | 控制信号传递         | 复杂的初始化流程，一步接一步       |
| 分层状态机 | 父子嵌套   | 包含与被包含       | 事件冒泡与继承       | 复杂设备模式切换                   |
