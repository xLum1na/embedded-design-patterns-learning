# 硬件适配器模式

​	**它的核心作用就是充当一个“翻译官”或“中间件”，解决上层应用软件与底层硬件设备之间因接口不兼容而无法协同工作的问题。**

## 一、为何存在

* **硬件接口各异**：需要集成多个功能相似但接口不同的硬件设备。例如，两个温度传感器都能测量温度，但一个提供 `read_temp()` 接口，另一个却是 `get_temperature()`，且返回的数据格式（如整数 vs 浮点数）和单位（如摄氏度 vs 华氏度）也不同。
* **代码复用困难**：上层业务逻辑期望一个统一、稳定的硬件接口，但底层硬件可能因成本、供货等原因频繁更换。如果没有适配层，更换硬件就意味着要重写或大量修改上层代码，维护成本极高。
* **跨平台移植**：希望将一套成熟的上层应用代码移植到不同的微控制器（MCU）平台上，但不同MCU的外设寄存器、厂商HAL库函数名称和参数各不相同。

## 二、核心角色与工作原理

### 1、核心角色

* **目标接口 (Target Interface)**：这是上层应用（客户端）所期望的统一接口规范。它定义了应用需要调用的一组标准服务，例如 `init()`、`read_data()`、`write_data()`。

* **适配者 (Adaptee)**：指那些已经存在的、接口不兼容的底层硬件设备或其驱动程序。它们拥有我们需要的功能，但“语言”不通。

* **适配器 (Adapter)**：这是模式的核心，它是一个软件模块。适配器**实现了目标接口**，并在其内部**调用适配者的具体接口**，完成接口和数据格式的“翻译”与转换。

* **上层应用 (Client)**：使用目标接口来访问硬件的业务逻辑代码。它完全不知道底层硬件的差异，只与适配器提供的统一接口打交道。



### 2、工作流程

* 上层应用调用适配器提供的标准接口，如：sensor_read()。
* 适配器接收到请求，将其“翻译”成底层具体硬件能理解的指令，如调用i2c_master_write_byte()或读取特定的硬件寄存器。
* 适配器获取硬件返回的原始数据后，可能还需要进行数据格式转换（如将原始ADC值转换为带单位的物理量），然后将结果返回给上层应用。
