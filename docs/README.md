## 📊 嵌入式设计模式学习优先级

### 🔴 **第一优先级：必须掌握（日常开发天天用）**

| 模式             | 理由                                                         | 掌握标准                                   |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------ |
| **有限状态机**   | 嵌入式最核心的模式，任何带模式的逻辑（按键、协议、设备状态）都用得到 | 能用手写switch-case实现，能理解状态表优化  |
| **中断模式**     | 嵌入式的基础，不懂中断就不懂嵌入式                           | 知道ISR要短、知道怎么用信号量/队列延迟处理 |
| **看门狗模式**   | 产品可靠性必备，任何工业/车规产品都必须有                    | 知道喂狗时机，能区分软硬件看门狗           |
| **硬件代理模式** | 驱动开发的基础，封装硬件操作                                 | 能写出.h/.c分离的传感器驱动                |
| **队列模式**     | RTOS任务通信的核心                                           | 能用队列解耦生产者和消费者                 |
| **单例模式**     | 硬件管理器、日志系统必用                                     | 知道C语言中怎么用static实现                |

**学习建议**：先啃这6个，能覆盖日常开发80%的场景。

### 🟠 **第二优先级：进阶必备（复杂项目高频出现）**

| 模式                | 理由                                 | 掌握标准                     |
| ------------------- | ------------------------------------ | ---------------------------- |
| **观察者模式**      | 事件驱动架构的基础                   | 能实现回调函数注册机制       |
| **命令模式**        | Shell/CLI、按键处理的标准方案        | 能把函数调用封装成命令对象   |
| **策略模式**        | 算法切换（滤波、电源模式）的优雅方案 | 能用函数指针表实现策略切换   |
| **工厂方法**        | 驱动管理、设备识别的标准模式         | 能根据设备ID动态创建驱动对象 |
| **发布-订阅**       | IoT设备（MQTT）必须理解              | 知道和观察者模式的区别       |
| **双缓冲区**        | 数据采集（ADC、摄像头）的经典方案    | 能实现乒乓缓冲               |
| **临界区/守卫调用** | 资源共享保护                         | 知道何时关中断、何时用互斥锁 |

**学习建议**：完成第一优先级后，优先学这些。能让你从“能跑”到“写得好”。

### 🟡 **第三优先级：架构设计（系统架构师必备）**

| 模式           | 理由                                       | 掌握标准                           |
| -------------- | ------------------------------------------ | ---------------------------------- |
| **桥接模式**   | 协议与介质解耦（如协议可跑在串口或CAN）    | 能设计硬件抽象层                   |
| **模板方法**   | 驱动框架设计（所有设备都按固定流程初始化） | 能用回调函数实现骨架+可变步骤      |
| **组合模式**   | 设备树管理、复杂GUI                        | 能递归处理树形结构                 |
| **责任链模式** | AT指令解析、多级滤波                       | 能实现链式处理                     |
| **中介者模式** | 多模块协调（如网关设备）                   | 能避免网状依赖                     |
| **适配器模式** | 旧代码复用、接口转换                       | 能在不改动原代码的前提下接入新硬件 |

**学习建议**：当你开始设计框架、写中间件、做平台化时，这些模式就是你的武器库。

### 🟢 **第四优先级：特定场景（用到再学，不用不学）**

| 模式             | 适用场景                                      |
| ---------------- | --------------------------------------------- |
| **状态模式**     | 面向对象的复杂状态机（C++项目用得多）         |
| **装饰器模式**   | 需要动态添加功能（如串口加加密/日志）         |
| **迭代器模式**   | 遍历复杂数据结构（链表、树）                  |
| **备忘录模式**   | 需要撤销/恢复功能（配置界面、调试器）         |
| **解释器模式**   | 需要解析脚本/协议（自动化测试、Modbus解析器） |
| **访问者模式**   | 需要对对象结构执行批量操作（如批量挂起设备）  |
| **享元模式**     | 需要共享大量常量数据（字体库、正弦表）        |
| **原型模式**     | 需要克隆对象（C语言极少用）                   |
| **建造者模式**   | 构造极其复杂的配置结构体（极少见）            |
| **抽象工厂模式** | 需要创建整套配套组件（极少独立使用）          |

**学习建议**：了解概念即可，遇到具体需求时再深入。
